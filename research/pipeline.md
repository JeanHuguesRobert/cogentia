## Object and associated documents

### Object of this document

This document formalizes a working method that has recently become more explicit and operational within the broader Cogentia corpus.

The method may conveniently be called a **pipeline**, but this term must be handled carefully.

A conventional pipeline suggests a linear circuit:

```text
input → processing → output
```

The Cogentia method is not a rigid circuit. It is closer to a **packet-switched cognitive network**:

```text
intuition
→ fragments
→ cognitive packets
→ versions
→ critiques
→ transformations
→ source documents
→ derived products
→ public feedback
→ reintegration into the corpus
→ new continuations
```

The term **pipeline** is retained for operational readability, but it should not be interpreted as an industrial, linear, closed workflow.

Core formulation:

> In Cogentia, the pipeline is not a production line.  
> It is a transformation network for cognitive packets.

Or, more compactly:

> **Pipeline on the surface, packet network in depth.**

## Abstract

This document defines the **Cogentia Pipeline** as a packet-based method for transforming latent possibilities into structured documents, reusable cognitive packets, and public-facing derived products.

The word “pipeline” is used as an operational convenience. The actual method is closer to packet switching than circuit switching: ideas are not processed through one linear channel; they circulate as cognitive packets, are routed through critique, recombined, versioned, embodied in source documents, transformed into derived products, published, and reintegrated into the corpus.

The purpose is not merely to produce texts faster. It is to create a living cognitive infrastructure in which ideas can be explored, tested, transformed, criticized, published, and reused without losing their genealogy.

---

## 1. Why “pipeline” is both useful and dangerous

The word **pipeline** is useful because it names an operational sequence:

```text
explore → stabilize → critique → package → derive → publish → reintegrate
```

It helps answer a practical question:

> How does an intuition become a source document, then a public-facing product, then a reusable part of the corpus?

However, the word is dangerous because it may imply a linear, deterministic, circuit-like process.

A conventional pipeline evokes:

- one input;
- one direction;
- one channel;
- one sequence;
- one output;
- minimal branching;
- little feedback.

Cogentia requires the opposite:

- multiple inputs;
- branching;
- partial outputs;
- critique loops;
- packets;
- continuations;
- recombination;
- derived products;
- re-entry into the corpus.

Thus the term must be redefined.

> A Cogentia pipeline is not a pipe.  
> It is a protocol for routing, transforming, recombining, publishing, and reintegrating cognitive packets.

---

## 2. Circuit logic vs packet logic

The distinction between circuit logic and packet logic is central.

### Circuit logic

Circuit logic assumes a dedicated channel between origin and destination.

Applied to cognition, it suggests:

```text
idea → draft → final publication
```

This model is simple, but often misleading. It treats the document as a single line of progression, as if the final text were the natural endpoint of the initial idea.

Its risks:

- premature closure;
- weak traceability;
- hidden transformations;
- loss of alternatives;
- overidentification of one form with the underlying idea;
- weak reuse.

### Packet logic

Packet logic treats ideas as packets that can be routed, copied, recombined, versioned, criticized, and reassembled.

Applied to Cogentia, it suggests:

```text
idea fragment
→ cognitive packet
→ critique
→ transformation
→ source document
→ derived products
→ feedback
→ corpus reintegration
```

The same cognitive packet may appear in:

- a research note;
- a Substack essay;
- a Facebook post;
- a YAML block;
- a political slogan;
- a GitHub issue;
- a future software tool;
- a public speech;
- an academic paper.

The packet does not belong to one document. It belongs to the corpus.

## 3. The general Cogentia Pipeline

The current version of the method can be represented as follows:

```text
1. Latent possibility
2. Exploratory conversation
3. Provisional formulation
4. Source document draft
5. Critical ping/pong
6. Transformation map
7. Cognitive packets
8. Objections and levels of evidence
9. Self-evaluation
10. Clean working paper
11. Derived products
12. Public publication
13. Feedback
14. Corpus reintegration
15. Continuation
```

Each stage can be skipped, repeated, split, or recombined. The sequence is therefore not a rigid chain.

It is a **default route**, not a fixed circuit. The stages are operational handles, not a bureaucratic checklist.

## 4. Stages, in compact form

### 4.1 Latent possibility

A latent possibility is not yet a thesis. It may be an intuition, phrase, synchronicity, contradiction, metaphor, political opportunity, historical analogy, or practical need.

Examples:

- “God-above vs God-beside”;
- “pipeline may be too linear”;
- “the Holy Spirit as spirit of synthesis”;
- “Autonomy of Capacity rather than paper autonomy”;
- “machine to explore vs machine to prevent”.

The first task is not to validate the idea. It is to preserve it long enough to explore its possible transformations.

### 4.2 Exploratory conversation

The latent possibility is tested through conversation with AI agents, human interlocutors, prior corpus documents, historical sources, or practical constraints.

The goal is not immediate closure. The goal is to expand the possibility space.

Useful operations:

- reformulation;
- analogy;
- objection;
- comparison;
- naming;
- decomposition;
- source-path identification.

### 4.3 Provisional formulation

A good provisional formulation is short, contestable, reusable, structurally clear, and not yet overcommitted.

Example:

> The word “pipeline” is retained for operational readability, but the actual method is closer to a packet-switched cognitive network.

### 4.4 Source document draft

A source document is the stable, versioned artifact that carries the deep structure of an idea.

It should normally include:

- frontmatter;
- object and associated documents;
- abstract;
- transformation map;
- main argument;
- objections;
- levels of evidence;
- cognitive packets;
- self-evaluation;
- continuation;
- references.

The source document is not necessarily the most readable public version. It is the corpus anchor.

### 4.5 Critical ping/pong

Once a draft exists, it is submitted to critique.

A common pattern is:

```text
draft
→ Grok critique
→ revision
→ ChatGPT restructuring
→ Grok critique
→ revision
→ publication candidate
```

The point is not to ask an AI whether the author is “right.” The point is to use externalized critique as a pressure test.

### 4.6 Transformation map

A transformation map documents how the idea evolved.

It answers:

> How did this idea become this document?

It prevents the final document from appearing as if it had emerged fully formed.

### 4.7 Cognitive packets

A cognitive packet is a compact, reusable unit of meaning.

It may contain:

- an identifier;
- title;
- type;
- definition;
- contrasts;
- use cases;
- risks;
- related packets;
- status.

Cognitive packets allow the corpus to behave less like a library of isolated documents and more like a living network of reusable structures.

### 4.8 Objections and levels of evidence

Every strong document should distinguish:

- facts;
- interpretations;
- hypotheses;
- analogies;
- sources of inspiration;
- rhetorical devices.

Basic levels:

```text
Level A — Established facts
Level B — Defensible interpretations
Level C — Corpus-specific hypotheses
Level D — Sources of inspiration
```

This prevents the corpus from collapsing into assertion.

### 4.9 Self-evaluation

The document should evaluate itself.

This is not a decorative gesture. It is an anti-capture mechanism.

Possible criteria:

- clarity;
- contestability;
- evidence separation;
- first-class objections;
- machine-readability;
- traceability;
- anti-capture;
- reusability;
- corpus integration;
- continuation.

A provisional “bullshit meter” can be used, provided it remains humble and explicit.

### 4.10 Clean working paper

A clean working paper should be good enough to:

- publish in a GitHub repository;
- be cited internally;
- be criticized externally;
- generate derived products;
- serve as a source for future work.

### 4.11 Derived products

A derived product is not a degradation of the source document.

It is a transformation of form for a specific audience, platform, or use case.

Examples:

- Substack article;
- Facebook post;
- Instagram caption;
- LinkedIn note;
- speech;
- academic abstract;
- slide deck;
- campaign leaflet;
- YAML packet.

The key distinction:

> The source document carries the fund.  
> The derived product adapts the form.

### 4.12 Publication, feedback, reintegration

Publication is not the end of the process. It is one routing event.

Feedback may include comments, AI critique, historical corrections, legal constraints, rhetorical failures, new analogies, objections, public reception, or platform constraints.

Reintegration may produce:

- a new version;
- a correction note;
- a new packet;
- a derived product;
- a separate working paper;
- a continuation item;
- a deprecated claim.

### 4.13 Continuation

Continuation is the explicit preservation of unfinished work.

It prevents premature closure and allows future agents or humans to resume work without rediscovering the whole context.

### 4.13.1 Issues as continuation packets

A GitHub Issue can function as a concrete continuation packet — distinct from both an in-flight `cogentia.continuation.*` object and a stabilized source document. It is opened when a question, objection, proposed transformation, missing source, or future task is stable enough to be tracked, but not yet ready to become a source-document change.

```text
conversation fragment
→ issue (memory in tension)
→ discussion / critique / evidence gathering
→ pull request or direct commit (proposed transformation)
→ source document update (stabilized memory)
→ issue closure with commit reference
```

The minimal grammar is:

```text
Issue          = memory in tension
Pull request   = proposed transformation
Commit         = stabilized memory
Source document = corpus anchor
```

A well-formed Cogentia Issue should carry, explicitly or implicitly:

- target document(s);
- problem or opportunity;
- proposed change (or, at minimum, the question);
- evidence level;
- risks of over-interpretation;
- expected closure condition;
- possible derived products;
- an agent-resumable next step.

The current `cogentia.js` v2 surface exposes a deliberately narrow read-only issue adapter:

- `cogentia issues list <repo>` — open / closed issues of a registered repo, pull requests filtered out;
- `cogentia issues packet <repo> <number>` — exports an issue as a typed packet (`cogentia.issue_continuation.v1`, YAML frontmatter + Markdown body), ready to be ingested by an agent or pasted into the next continuation.

Consolidation note, 2026-06-17: this adapter is read-only and GitHub-specific. It does not close, label, comment on, or otherwise mutate issues. That preserves the doctrine: GitHub Issues can be memory packets for continuation, but they do not become the only possible work-item backend and they do not replace human/DHITL arbitration.

Minimal formula:

> Issue = unresolved continuation made addressable.

### 4.14 README maintenance

A repository README is not only documentation. On GitHub it is the public dashboard of a repository — or, in a subdirectory, of a subsystem. As the corpus grows, these dashboards drift out of date.

The pipeline therefore includes an **opt-in README maintenance** step. A README, at the repository root or in any subdirectory, may carry a marked block:

```text
<!-- BEGIN_AUTO: readme_index -->
<!-- END_AUTO: readme_index -->
```

Earlier designs named this operational tool `cogentia readme`, also run as part of `cogentia refresh`. The current v2 CLI does not expose that command. The intended behavior remains: regenerate, inside that block only, an index of the Markdown documents living in the README's own subtree. A README without the marker is never modified — which keeps hand-written public pages, including the public profile README, fully under human control.

This is a derived-view step in the same family as backlinks, trails and documents: it transforms source files into a navigable public surface without overwriting curated prose. It embodies the same default as the rest of the pipeline — *publicity by default, explicit and bounded automation, human control of the page itself.*

**The user-attached profile README is treated apart.** The root README of the profile repository (the `github.com/<user>/<user>` page) is not a mechanical index but a **derived product** (§4.11) — a public, human-facing page authored *from* the corpus sources. The pipeline therefore does **not** auto-generate it. The intended maintenance route is a typed **continuation** (`cogentia.continuation.v2` in the current CLI) that delegates the refresh to the intelligent agent: the continuation cites the source documents (e.g. `research/index.md`, `research/agent_brief.md`, `CONTEXT.md`, `POSSIBILISM.md`, `PROJECTS.md`, `TIMELINE.md`) and asks the agent to rewrite the page faithfully, citing its sources and inventing nothing. The delegation should be idempotent — at most one active continuation per profile README — and the agent itself decides whether a refresh is actually warranted. This keeps a mechanical tool from authoring a human's public identity, while still scheduling its upkeep inside the pipeline. It is a small but exact illustration of the source ↔ derived split: the index block is symmetric and machine-generated; the profile page is an asymmetric derived product that requires judgment, so it is routed to an agent rather than a regex.

**The same routing applies to a growing family of derived products.** Auto-generated tutorials, curated reading trails, and rendered websites should likewise be refreshed by grouped continuations rather than mechanical rewriting. Earlier prototypes named this command `cogentia derived` inside `refresh`; the current v2 CLI exposes only generic continuations, so the grouped scheduler remains to be reintroduced. Detection is hybrid: a document opts in with frontmatter `derived_by: agent`; trails are `research/trails/*.md` by convention; websites are any directory containing `_config.yml` (repository root or subdirectory). Each grouped continuation lists its items and asks the agent to regenerate them faithfully from their declared sources, citing them and inventing nothing — the agent deciding, per item, whether a refresh is actually warranted. The **typology** of these delegated derived products (profile README, tutorials, trails, websites, …) is maintained in [`derived_products.md`](derived_products.md); this section describes only how the pipeline schedules their upkeep.

Taken together — mechanical views that regenerate themselves, and judgment-requiring products that are delegated and refreshed on demand — these artefacts define the intended **reactive corpus** loop: documents, websites, and other rendered surfaces stay live by reacting to changes in their sources rather than silently drifting. Consolidation note, 2026-06-09: this loop is doctrinally stable but only partially implemented in the current v2 CLI. Mechanical corpus verification is active; grouped derived-product scheduling still needs to be restored or redesigned.

## 6. Case studies

The pipeline should not remain a purely abstract method. It becomes more useful when tested against recent corpus artifacts.

### 6.1 Full case study — `christianity_verticalization.md`

The working paper [`christianity_verticalization.md`](https://github.com/JeanHuguesRobert/barons-Mariani/blob/main/research/christianity_verticalization.md) is the clearest recent example of the method operating end-to-end.

#### Trigger

The trigger was not an academic research question in the narrow sense. It was a symbolic chain:

```text
Pentecost
→ Psalm 23
→ shepherd / flock
→ donkey
→ Jesus entering Jerusalem
→ God-beside vs God-above
→ verticalization of Christianity
```

The initial trigger was not treated as proof. It was treated as an entry point into structured exploration.

#### Exploratory phase

The conversation clarified a symbolic distinction:

```text
God-above
→ shepherd
→ flock
→ guidance from above
→ promised pastures
→ political verticality
```

versus:

```text
God-beside
→ companion
→ donkey
→ shared path
→ citizens capable
→ political accompaniment
```

At this stage, the idea could still have become merely rhetorical or mystical. The pipeline forced it into a more auditable structure.

#### Source document

A first Markdown source document was generated with:

- frontmatter;
- object and associated documents;
- historical development;
- objections;
- levels of evidence;
- continuation;
- method note.

This established a corpus anchor.

#### Critical ping/pong

Grok evaluated the draft according to the second method and identified missing or weak elements:

- lack of a transformation map;
- lack of cognitive packets;
- bibliography still indicative;
- need for stronger self-evaluation;
- risk of essayistic style.

These critiques became routing instructions for the next version.

#### Packetization and later versions

The next versions added:

- transformation map;
- YAML cognitive packets;
- self-evaluation;
- explicit bullshit meter;
- section on Pentecost;
- section on Mary, Mother of the Church;
- section on the Holy Spirit as spirit of synthesis;
- AI-assisted genesis note.

The document therefore moved from essay to source artifact.

#### Derived product

A Substack blogpost was produced:

> “Dieu au-dessus, Dieu à côté : pourquoi l’âne compte politiquement”

The derived product did not replace the source.

```text
source document = deep structure, objections, packets, evidence levels
blogpost = narrative accessibility, public circulation, symbolic force
```

#### Public sharing and reintegration

The Substack article generated platform-specific sharing drafts:

- Facebook;
- Instagram;
- possible LinkedIn adaptation.

The process revealed the need to formalize the method itself. This led directly to the present `pipeline.md` (this document).

Essential lesson:

> A successful pipeline does not only produce documents.  
> It can produce awareness of the method that produced them.

### 6.2 Full case study — `alien_academic`

The `alien_academic` workflow represents a second mature example, because it begins from a cultural artifact rather than from an explicitly political or theological concept.

#### Trigger

The starting point was a popular cultural object: *Alien*.

The film was not used as evidence. It was used as a cognitive trigger.

The danger was obvious:

> confusing a vivid metaphor with an argument.

The second method prevented that by forcing the metaphor through structured transformation.

#### Extraction

The cultural object was used to extract a deeper structure:

```text
Alien / xenomorph
→ non-human intelligence
→ overconfidence
→ unknown agency
→ fragile human assumptions
→ AI safety / epistemic humility
```

The film provided the hook. The source document had to provide the argument.

A weaker process would have produced only:

```text
Alien is like AI.
```

The stronger process produced:

```text
A cultural artifact can expose a structural failure mode:
human overconfidence in the face of opaque non-human agency.
```

#### Source stabilization

The idea was transformed into an academic-style working paper with:

- abstract;
- conceptual framing;
- distinctions;
- objections;
- self-application;
- critique;
- revision.

This is where the pipeline protects against pure analogy.

#### Critical revision

Grok critique and subsequent revisions strengthened:

- argumentative discipline;
- self-evaluation;
- distinction between metaphor and proof;
- transition from academic source to public essay.

This case shows why critical ping/pong matters. It prevents the author from falling in love with the first elegant formulation.

#### Derived products

The source then produced:

- a Substack version;
- social sharing drafts;
- platform-specific adaptations.

Again, the fund remained in the source. The form changed according to the audience.

#### Methodological lesson

The `alien_academic` case shows that the pipeline can start from almost anything:

- a film;
- a synchronicity;
- a political event;
- a technical analogy;
- a biblical image;
- a practical administrative problem.

But the trigger is not the thesis. The pipeline exists precisely to transform triggers into structured, contestable, reusable artifacts.

### 6.3 Case study — `derived_products.md`

The document [`derived_products.md`](derived_products.md) provides the typology needed by the pipeline.

It clarifies that a derived product is not a mere simplified copy. It is a transformation of form relative to a source.

```text
source document = fund, structure, traceability
derived product = form, audience, platform, circulation
```

This makes the corpus generative rather than static.

## 8. Objections and safeguards

### Objection 1 — “Pipeline” is the wrong word

This objection is valid.

The word “pipeline” suggests a linear channel. The Cogentia method is not linear.

Response:

> The term is retained for operational convenience, but explicitly redefined as a packet-based transformation network.

The ambiguity is not hidden. It is documented and becomes part of the method.

### Objection 2 — The method could become bureaucratic

A method designed to preserve traceability may become heavy.

Risk:

- too many required sections;
- excessive formalism;
- slower publication;
- loss of spontaneity.

Response:

> The pipeline is a default route, not a mandatory checklist.

For small ideas, a short packet may be enough. For major claims, a full source document is preferable.

### Objection 3 — A pipeline document may become a donkey staring at its own reflection

A document explaining its own method can easily become self-referential.

In less technical language:

> A pipeline document explaining its own pipeline may become an elegant donkey staring at its reflection.

This is not only a stylistic risk. It is a methodological risk.

A meta-document can become a machine for admiring its own coherence instead of producing usable outputs.

Response:

- keep the jokes modest but visible;
- keep the packets reusable;
- keep the objections first-class;
- keep external references;
- keep practical continuations;
- test the method on actual documents;
- accept that all elegant systems should occasionally be pulled back to the muddy path.

The purpose of the second method is precisely to prevent self-description from becoming self-celebration.

### Objection 4 — AI-assisted ping/pong could become circular validation

If ChatGPT drafts and Grok praises, the process may create an illusion of external validation.

Response:

- AI critique must not be treated as authority;
- objections must be first-class;
- sources must be checked;
- author responsibility must remain explicit;
- human review remains valuable;
- publication invites external correction.

### Objection 5 — Derived products may distort the source

A Substack article, Facebook post or slogan may oversimplify the source.

Response:

- derived products should point back to the source when possible;
- symmetric vs asymmetric derived products should be distinguished;
- risky condensations should be documented;
- source documents remain the corpus anchor.

### Objection 6 — Packetization may fragment thought

Breaking ideas into packets may encourage modularity at the expense of depth.

Response:

> Packets are not replacements for source documents.  
> They are reusable handles for ideas embedded in richer documents.

The corpus needs both:

- packets for routing and reuse;
- documents for depth and continuity.

## 13. References and external anchors

This method note is primarily internal to the Cogentia corpus, but it is informed by several external traditions. These references should not be read as claims of equivalence. They are anchors and analogies.

### 13.1 Packet switching and distributed communication

- Paul Baran, *On Distributed Communications: I. Introduction to Distributed Communications Networks*, RAND Corporation, RM-3420-PR, 1964.  
  Useful for thinking about survivable distributed networks, redundancy and routing without a single fragile center.

- Donald Davies and the National Physical Laboratory packet switching work, 1965–1967.  
  Useful for the terminology and practical emergence of packet switching as an alternative to dedicated circuit logic.

Methodological relevance:

```text
circuit switching → fixed channel, dedicated path, fragile centrality
packet switching → distributed routing, recombination, resilience
```

Cogentia uses this as an analogy for cognition and corpus work, not as a literal networking model.

### 13.2 Agile and reactive systems

- Kent Beck et al., *Manifesto for Agile Software Development*, 2001.  
  Useful as an external anchor for iterative, adaptive, artifact-oriented work.

- The Reactive Manifesto, 2014.  
  Useful for the emphasis on responsiveness, resilience, elasticity and message-driven systems. In this document, the author’s personal recollection of having signed or supported it is marked as requiring verification.

Methodological relevance:

```text
agile → iteration, feedback, working artifacts
reactive → responsiveness, resilience, message-driven interaction
Cogentia → cognitive packets, critique loops, continuation
```

### 13.3 Literate programming and source artifacts

- Donald E. Knuth, “Literate Programming,” *The Computer Journal*, Volume 27, Issue 2, 1984, pp. 97–111. DOI: `10.1093/comjnl/27.2.97`.  
  Useful for thinking about documents as artifacts addressed to human understanding, not merely machine execution.

Methodological relevance:

```text
code should explain itself to humans
source documents should explain their reasoning to future readers and agents
```
